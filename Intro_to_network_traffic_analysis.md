# Intro to Network Traffic Analysis #



Common traffic analysis tools
-----------------------------

tcpdump   => command-line ပေါ်ကနေ network traffic ကို capture လုပ်ပြီး ကြည့်နိုင်တဲ့ tool ပါ။

tshark    => tcpdump လိုပဲ command-line packet analyzer ဖြစ်ပါတယ်။

wireshark => GUI ပါတဲ့ network traffic analyzer ဖြစ်ပြီး NTA မှာ အရေးအကြီးဆုံး tool တစ်ခုပါ။

NGrep     => Linux ရဲ့ grep လိုပဲ pattern matching လုပ်တဲ့ tool ဖြစ်ပေမယ့် Network traffic ပေါ်မှာ အသုံးပြုပါတယ်။

tcpick    => TCP streams ကို track လုပ်ပြီး ပြန်လည်စုစည်း (reassemble) လုပ်နိုင်တဲ့ tool ပါ။

Network taps => hardware devices, Network traffic ကို copy လုပ်ပြီး analysis system ကို ပို့တယ်, Original traffic ကို မထိခိုက်ဘဲ passively capture လုပ်နိုင်

Networking span ports => SPAN (Port Mirroring) ကSwitch / Router မှာ traffic ကို copy လုပ်ပြီးတခြား port တစ်ခုကို ပို့ပေးတဲ့ နည်းလမ်း

Elastic Stack => Traffic နဲ့ logs တွေကိုIngest,Store,Visualize,Search လုပ်နိုင်တဲ့ tool set ပါ။

SIEM => SIEM (Security Information and Event Management) က Security data အားလုံးကို စုစည်းတဲ့ central system ဖြစ်ပါတယ်။




Berkeley Packet Filter (BPF)
----------------------------

BPF ဆိုတာ Network Traffic တွေကို စစ်ဆေးတဲ့ Tool တော်တော်များများမှာ အသုံးများတဲ့ Syntax (ရေးထုံး) တစ်ခုဖြစ်ပါတယ်။Network ရဲ့ Data-Link Layer (Layer 2) ကနေ ဒေတာတွေကို တိုက်ရိုက်ဖတ်နိုင်၊ ရေးနိုင်အောင် လုပ်ဆောင်ပေးတဲ့ နည်းပညာတစ်ခုလည်း ဖြစ်ပါတယ်။




Performing Network Traffic Analysis
-----------------------------------

1. Simple Analysis => Terminal / Console ပေါ်မှာ Live traffic ကိုကြည့်နေခြင်းနဲ့တင် Analysis လုပ်နိုင်ပါတယ်။ (tcpdump -i eth0)
2. Advanced Analysis => Network tap နဲ့ traffic ကို capture, Captured traffic ကို SIEM ဆီပို့, SIEM မှာ
    Signature matching
    Alert detection
    TTPs (Tactics, Techniques) ခွဲခြမ်းစိတ်ဖြာ
3. Passive listening => Traffic ကို passively နားထောင်ဖမ်းယူဖို့အနည်းဆုံး လိုအပ်တာက ကိုယ်ဖမ်းချင်တဲ့ network segment ထဲကို တိုက်ရိုက်ချိတ်ထားရမယ်
4. Switched Environment => most today's network is switched environment, VLAN, Switch Ports တွေကBroadcast domain မတူရင် traffic ကို forward မလုပ်ပါဘူး ဒါကြောင့် ကိုယ့် device ကအဲ့ဒီ VLAN ထဲ မရှိဘူးဆိုရင် traffic မမြင်ရပါဘူး





Network Traffic Analysis workflow
---------------------------------

![](/home/zane001/Pictures/Screenshots/workflow.png)

**NTA Workflow** ဆိုတာNetwork Traffic Analysis ကို စနစ်တကျလုပ်ဆောင်ဖို့အဆင့်လိုက် ပြန်လည်လုပ်ဆောင်နေတဲ့ cycle (loop) တစ်ခုပါ။



**1. Ingest traffic** 

Capture location ကို ရွေးချယ်ပြီးTraffic ကို စတင်ဖမ်းယူပါ မိမိရှာဖွေနေတာကို ကြိုတင် သိထားရင် **Capture Filters (BPF)** ကို အသုံးပြုပါ



**2. Reduce Noice by Filtering**

Production network တစ်ခုရဲ့ link ကို capture လုပ်ရင်Traffic အလွန်များ, Noise အလွန်ကြီး ဖြစ်တတ်ပါတယ်။ဒါကြောင့်Capture ပြီးရင် မလိုအပ်တဲ့ traffic ကို filter လုပ်ပြီးဖယ်ရှားပါ



**3. Analyze and Explore**

ဒီအဆင့်မှာကိုယ်ရှာဖွေနေတဲ့ ပြဿနာနဲ့ သက်ဆိုင်တဲ့ data ကို တိတိကျကျ ရွေးထုတ်ပါတယ်။စစ်ဆေးရမယ့်အရာတွေ

Host တစ်ခုချင်း, Protocol တစ်မျိုးချင်း, TCP header flags (SYN, ACK, FIN, RST)

ကိုယ့်ကိုယ်ကို မေးသင့်တဲ့ မေးခွန်းများ -

Traffic က

Encrypted လား Plaintext လား?

Encrypted ဖြစ်သင့်လား?

User တွေကမရသင့်တဲ့ resource ကို ဝင်ရောက်ဖို့ ကြိုးစားနေလား?

ပုံမှန် မဆက်သွယ်တဲ့ hosts တွေအချင်းချင်း ဆက်သွယ်နေလား?



**4. Detect and Alert**

ဒီအဆင့်မှာ

Errors တွေ ရှိလား?

Response မပေးတဲ့ device ရှိလား?

ပုံမှန်မဟုတ်တဲ့ pattern တွေ ရှိလား?  

တွေကို စစ်ဆေးပါတယ်။

ဒီ traffic က Benign (အန္တရာယ်မရှိ) လား, Malicious (အန္တရာယ်ရှိ) လား ဆုံးဖြတ်ပါ

ဒီအဆင့်မှာ IDS / IPS, Signature-based detection, Heuristic analysis တို့ကို အသုံးပြုလို့ရပါတယ်။



**5. Fix and Monitor**

ဒီအဆင့်ကNTA cycle ထဲမှာ မပါတဲ့အဆင့် ဖြစ်ပေမယ့်အရေးကြီးဆုံးအဆင့် တစ်ခုပါ။

Issue ကို fix လုပ်ပြီးနောက် အဲ့ဒီ source / host / link ကို အချိန်တစ်ခုအထိ ဆက်လက် monitor လုပ်ပါ 

ပြဿနာ တကယ်ပြီးဆုံးသွားပြီလား Attack ပြန်လာမလာဆိုတာကို သေချာစစ်ဖို့ပါ။

## Networking Primer - Layers 1-4 ##



**OSI / TCP-IP Models **

![](/home/zane001/Pictures/Screenshots/net_models4.png)

**Model Traits Comparison (OSI vs TCP/IP)**

| အချက်           | **OSI Model**                        | **TCP/IP Model**                          |
| --------------- | ------------------------------------ | ----------------------------------------- |
| Layers အရေအတွက် | 7 Layers                             | 4 Layers                                  |
| Flexibility     | တင်းကျပ် (Strict)                    | ပိုလွတ်လပ် (Loose)                          |
| Dependency      | Protocol-independent (သဘောတရားအခြေခံ) | Common communication protocols အပေါ်အခြေခံ |

**OSI Model**

Network အလုပ်လုပ်ပုံကိုသဘောတရား (theoretical) အနေနဲ့ ခွဲခြမ်းဖော်ပြထားတဲ့ model ဖြစ်ပါတယ်။



**Layer 1 – 4 (Data Transport အပိုင်း)**

ဒီ layers တွေက

- Physical medium (ကြိုး၊ Wi-Fi)
- Frame / Packet / Segment
- Protocol control (TCP/UDP)

တွေကို စီမံခန့်ခွဲပါတယ်။



**Layer 5 – 7 (Data Interpretation အပိုင်း)**

ဒီ layers တွေက

- Session management
- Data format / encryption
- Application interaction

တွေကို ကိုင်တွယ်ပါတယ်။





**TCP/IP Model **

TCP/IP Model ကလက်တွေ့ networking မှာ အမှန်တကယ် အသုံးပြုနေတဲ့ model ဖြစ်ပါတယ်။



**TCP/IP Layer 4 – Application Layer**

OSI Layer 5, 6, 7 ကို ပေါင်းထားတယ် 

HTTP, HTTPS, FTP, SMTP, DNS

User နဲ့ တိုက်ရိုက် ဆက်သွယ်တဲ့အပိုင်း



**TCP/IP Layer 3 – Transport Layer**

OSI Layer 4 နဲ့ တူ

TCP / UDP

Data transport ကို ထိန်းချုပ်



**TCP/IP Layer 2 – Internet Layer**

OSI Layer 3 နဲ့ ကိုက်ညီ

IP, ICMP

Routing, IP addressing



**TCP/IP Layer 1 – Link Layer**

OSI Layer 1 & 2 ကို ပေါင်းထား

Ethernet, ARP, MAC addressing

Physical & Data link အပိုင်း

## PDU Example ##

**PDU** ဆိုတာ
 OSI Model ရဲ့ layer တစ်ခုချင်းစီက Control information (headers)Data (payload)တွေကို encapsulate လုပ်ထားတဲ့
data packet တစ်ခု ဖြစ်ပါတယ်။Layer တစ်ခုစီမှာ PDU အမည် မတူပါဘူး



![](/home/zane001/Pictures/Screenshots/net_models_pdu2.png)

PDU (Protocol Data Unit) ကို စစ်ဆေးတဲ့အခါEncapsulation (အလွှာလိုက် ထုပ်ပိုးခြင်း) ဆိုတဲ့ အယူအဆကိုမဖြစ်မနေ ထည့်စဉ်းစားရပါမယ်။

**Encapsulation**

Data ကProtocol stack ကို အပေါ်ကနေ အောက်ကို ဆင်းသွားတဲ့အခါ Layer တစ်ခုချင်းစီက အောက်က layer ကို မပို့ခင် အပေါ် layer ရဲ့ data ကို ကိုယ့် layer ရဲ့ header နဲ့ ထပ်ထုပ်ပိုးပေးခြင်း ကိုEncapsulation လို့ ခေါ်ပါတယ်။

## MAC Addressing ##

![](/home/zane001/Pictures/Screenshots/macAddressing.png)

MAC address မှာ အပိုင်း ၆ ပိုင်း (Octets) ပါဝင်ပြီး စုစုပေါင်း 48-bit ရှိပါတယ်။ဥပမာ - `00:1A:2B:3C:4D:5E`

ကွန်ရက်တစ်ခုတည်းမှာရှိတဲ့ Host အချင်းချင်း ဆက်သွယ်ဖို့အတွက် MAC address ကို သုံးပါတယ်။ဒေတာတစ်ခုဟာ Router တစ်ခုကနေ တစ်ခုကို ဖြတ်သွားတိုင်း Source နဲ့ Destination **MAC addresses တွေဟာ ပြောင်းလဲသွားပါတယ်** (IP address ကတော့ မပြောင်းလဲပါ)။

- Router တွေဟာ ဒေတာတွေကို ရှေ့ဆက်ပို့ဖို့အတွက် Layer 2 Header တွေကို အမြဲတမ်း ခွာထုတ်ပြီး အသစ်ပြန်လဲပေးရပါတယ်။

## IP Addressing ##

**IP (Internet Protocol)** ဆိုတာ ဒေတာတွေကို ကွန်ရက်တစ်ခုကနေ နောက်တစ်ခုဆီ (Cross-network boundaries) ရောက်ရှိအောင် ပို့ဆောင်ပေးဖို့ တီထွင်ထားတဲ့ စနစ်ဖြစ်ပါတယ်။ IP ဟာ Packet တွေကို လမ်းကြောင်းပြခြင်း (**Routing**), ဒေတာတွေကို ထုပ်ပိုးခြင်း (**Encapsulation**), ဒေတာတွေကို ခွဲစိတ်ခြင်းနှင့် ပြန်လည်ပေါင်းစည်းခြင်း (**Fragmentation and Reassembly**) စတာတွေကို တာဝန်ယူရပါတယ်။

**အဓိက Version များ:** လက်ရှိမှာ IP Version နှစ်မျိုးရှိပါတယ်။

**IPv4:** လက်ရှိ ကမ္ဘာပေါ်မှာ အသုံးအများဆုံး စံနှုန်းဖြစ်ပါတယ်။

**IPv6:** IPv4 နေရာမှာ အစားထိုးဖို့ ပေါ်လာတဲ့ မျိုးဆက်သစ် စံနှုန်းဖြစ်ပါတယ်။



**IPv4**

![](/home/zane001/Pictures/Screenshots/ipv4Addressing.png)

IPv4 လိပ်စာတစ်ခုမှာ စုစုပေါင်း 32-bit ပါဝင်ပါတယ်။အဲဒီ 32-bit ကို 8-bit စီပါတဲ့ အပိုင်း ၄ ပိုင်း (Octets) ခွဲထားပါတယ်။

လူတွေဖတ်ရလွယ်အောင် ဒသမကိန်း (Decimal) နဲ့ ရေးသားပါတယ်။ ဥပမာ - `192.168.86.243`

Octet တစ်ခုချင်းစီမှာ ဂဏန်းဟာ `0` ကနေ `255` အတွင်းမှာပဲ ရှိရပါမယ်။Network Traffic Analysis လုပ်တဲ့အခါ IPv4 ကို 

In OSI model => Layer 3 (Network Layer)

In TCP/IP model => Layer 2 (Internet Layer)



**IPv6**

![](/home/zane001/Pictures/Screenshots/ipv6Addressing.png)

IPv6 သည် ကမ္ဘာပေါ်ရှိ မည်သည့် Network ပစ္စည်းအတွက်မဆို လုံလောက်သော Address Space ကို ပေးစွမ်းနိုင်သည်။

IPv4 နေရာတွင် အစားထိုးရန် တီထွင်ထားသော Successor ဖြစ်သည်။

 IPv4 ထက် ပိုမိုကောင်းမွန်သော Routing စနစ်နှင့် လုံခြုံရေး (Built-in IPsec) တို့ ပါဝင်လာသည်။

Address ပေါများသည့်အတွက် NAT (Network Address Translation) ကို အသုံးမပြုဘဲ စက်ပစ္စည်းအချင်းချင်း တိုက်ရိုက်ဆက်သွယ်နိုင်စွမ်း ပိုကောင်းလာသည်။

| **အမျိုးအစား (Type)** | **ဖော်ပြချက် (Description)**                                 | **အလွယ်မှတ်ရန်**                 |
| ------------------- | ------------------------------------------------------------ | -------------------------------- |
| **Unicast**         | Interface တစ်ခုတည်း (စက်တစ်လုံးတည်း) အတွက် လိပ်စာ။               | တစ်ဦးချင်း (1 to 1)              |
| **Anycast**         | Interface အများအပြားအတွက် လိပ်စာဖြစ်ပြီး၊ အနီးဆုံးရှိသော စက်တစ်ခုကသာ ဒေတာကို လက်ခံရရှိမည်။ | အုပ်စုထဲမှ တစ်ဦးတည်း (1 to nearest) |
| **Multicast**       | Interface အများအပြားအတွက် လိပ်စာဖြစ်ပြီး၊ အုပ်စုဝင်အားလုံး ဒေတာကို တစ်ပြိုင်နက် လက်ခံရရှိမည်။ | အများအပြား (1 to many)           |
| **Broadcast**       | IPv6 တွင် Broadcast မရှိတော့ပါ။ ၎င်းအစား Multicast ကိုသာ အသုံးပြုပါသည်။ | (မရှိတော့ပါ)                       |

## TCP / UDP, Transport Mechanisms



**Transport Layer** ကို ကွန်ရက်တစ်ခုရဲ့ **"ထိန်းချုပ်ရေးဗဟိုဌာန (Control Hub)"** လို့ တင်စားနိုင်ပါတယ်။ ၎င်းရဲ့ အဓိကတာဝန်ကတော့ အပေါ်ထပ် Layer (Application) ကနေ ဆင်းလာတဲ့ ဒေတာတွေကို အောက်ထပ် Layer (IP/MAC) တွေဆီ ပို့ဆောင်နိုင်အောင် ထုပ်ပိုးပေးခြင်း (**Encapsulation**) နဲ့ ဒေတာတွေ လက်ခံရရှိတဲ့အခါ မူလအတိုင်း အစီအစဉ်တကျ ပြန်လည်ပေါင်းစည်းပေးခြင်း (**Reassembling**) တို့ပဲ ဖြစ်ပါတယ်။



| **အင်္ဂါရပ်**   | **TCP (Transmission Control)**                   | **UDP (User Datagram)**                                 |
| -------------- | ------------------------------------------------ | ------------------------------------------------------- |
| **ပို့ဆောင်မှုပုံစံ** | **Connection-oriented:** သေချာချိတ်ဆက်ပြီးမှ ပို့သည်။ | **Connectionless:** ပို့ပြီးရင် မေ့လိုက်သည် (Fire and forget)။ |
| **ချိတ်ဆက်မှု**  | **3-way handshake** သုံးပြီး အရင်ချိတ်ဆက်သည်။        | တစ်ဖက်က အဆင်သင့်ရှိမရှိ မစစ်ဆေးပါ။                         |
| **ဒေတာပေးပို့မှု** | ဒေတာစီးကြောင်း (Stream) အလိုက် ပို့သည်။                | တစ်ထုပ်ချင်း (Packet by packet) ပို့သည်။                    |
| **အတည်ပြုချက်** | ဒေတာရောက်ကြောင်း အတည်ပြုချက် (Ack) ပို့ရသည်။         | အတည်ပြုချက် မလိုအပ်ပါ။                                     |
| **အမြန်နှုန်း** | စစ်ဆေးမှုများသောကြောင့် **နှေးသည်။**               | စစ်ဆေးမှုမရှိဘဲ တိုက်ရိုက်ပို့သောကြောင့် **မြန်သည်။**              |



##### TCP ကို ဘယ်အချိန်မှာ သုံးသလဲ? (Reliability over Speed)

**TCP** ကို ဒေတာတွေ တိကျမှန်ကန်ဖို့နဲ့ အကုန်အစင်ရောက်ရှိဖို့ လိုအပ်တဲ့အခါ သုံးပါတယ်။

- **ဥပမာ:** SSH ဖြင့် ကွန်ပျူတာတစ်လုံးကို လှမ်းချိတ်ပြီး Password ချိန်းတဲ့အခါ။
- **ဘာကြောင့်လဲ:** ဒေတာအချို့ လမ်းမှာ ပျောက်သွားရင် TCP က အဲဒီအပိုင်းကို ပြန်ပို့ပေးပါတယ်။ ဒါကြောင့် Password ပြောင်းတဲ့ Command ဟာ တစ်ဝက်တစ်ပျက်နဲ့ မှားယွင်းသွားတာမျိုး မဖြစ်အောင် ကာကွယ်ပေးပါတယ်။



##### UDP ကို ဘယ်အချိန်မှာ သုံးသလဲ? (Speed over Completeness)

**UDP** ကို ဒေတာအစုံအလင် ရောက်ဖို့ထက် အချိန်နဲ့တစ်ပြေးညီ မြန်မြန်ရောက်ဖို့ လိုအပ်တဲ့အခါ သုံးပါတယ်။

- **ဥပမာ:** Video Streaming (YouTube/Netflix) ကြည့်ခြင်း သို့မဟုတ် DNS Query လုပ်ခြင်း။
- **ဘာကြောင့်လဲ:** ဗီဒီယိုကြည့်တဲ့အခါ ပုံရိပ်လေး တစ်ချက်နှစ်ချက် ကွဲသွားတာ (Dropped pixels) ထက် ဗီဒီယိုကြီး ရပ်တန့် (Buffer) နေတာက ပိုဆိုးပါတယ်။ DNS လိုမျိုးမှာလည်း အဖြေမြန်မြန်ရဖို့က ပိုအရေးကြီးပြီး၊ ပျောက်သွားရင်လည်း နောက်တစ်ခါ ထပ်တောင်းလိုက်ရုံပါပဲ။

## TCP Three-way Handshake

![](/home/zane001/Pictures/Screenshots/three-way-handshake.png)

**SYN (Client → Server):**

- Client ကနေ Server ဆီကို "စကားပြောလို့ ရမလား" လို့ ခွင့်တောင်းတဲ့အနေနဲ့ SYN Packet ကို ပို့ပါတယ်။
- ဒီအဆင့်မှာ ဒေတာတွေကို နံပါတ်စဉ်တပ်ဖို့ **Sequence Number** ကို နှစ်ဖက်သဘောတူညီအောင် စတင်သတ်မှတ်ပါတယ်။ ဒါ့အပြင် Window size နဲ့ အခြား နည်းပညာဆိုင်ရာ အချက်အလက်တွေကိုပါ ညှိနှိုင်းပါတယ်။

**SYN-ACK (Server → Client):**

- Server က Client ရဲ့ တောင်းဆိုမှုကို လက်ခံရရှိတဲ့အခါ SYN နဲ့ ACK Flag နှစ်ခုစလုံး ပါတဲ့ Packet ကို ပြန်ပို့ပါတယ်။
- **ACK** က "မင်းပို့တဲ့ SYN ကို ငါရပြီ" လို့ ပြောတာဖြစ်ပြီး၊ **SYN** ကတော့ "ငါဘက်ကလည်း မင်းနဲ့ စကားပြောဖို့ အဆင်သင့်ပဲ၊ ငါ့ရဲ့ Sequence Number ကိုလည်း မှတ်ထားပေးပါ" လို့ ပြန်ညှိနှိုင်းတာ ဖြစ်ပါတယ်။

**ACK (Client → Server):**

- Client က Server ရဲ့ ပြန်ကြားချက်ကို လက်ခံရရှိကြောင်း အတည်ပြုတဲ့အနေနဲ့ ACK Flag ပါတဲ့ Packet ကို နောက်ဆုံးအကြိမ် ထပ်ပို့ပါတယ်။
- ဒီအဆင့်ပြီးသွားရင်တော့ Client နဲ့ Server ကြားမှာ ခိုင်မာတဲ့ ချိတ်ဆက်မှု (Established Connection) ရရှိသွားပြီး ဒေတာတွေကို စတင် ပေးပို့နိုင်ပြီ ဖြစ်ပါတယ်။

## TCP Session Teardown

![](/home/zane001/Pictures/Screenshots/session-teardown.png)

ဒေတာပေးပို့မှုတွေ အားလုံး ပြီးဆုံးသွားတဲ့အခါ TCP ဟာ ချိတ်ဆက်မှုကို စနစ်တကျ ပြန်လည်ပိတ်သိမ်းဖို့အတွက် **FIN (Finish)** flag ကို အသုံးပြုပါတယ်။ ဒါကို **Four-way Handshake** လို့လည်း ခေါ်ဝေါ်လေ့ရှိပါတယ်။



**FIN, ACK (Client → Server):**

- Client က ဒေတာပို့လို့ ပြီးသွားပြီဖြစ်ကြောင်းနဲ့ ချိတ်ဆက်မှုကို ရပ်နားချင်ကြောင်း ပြောဖို့ **FIN** flag ပါတဲ့ packet ကို ပို့ပါတယ်။ (ACK ကတော့ အရှေ့မှာရခဲ့တဲ့ ဒေတာတွေအတွက် အတည်ပြုချက်ပါ)။

**ACK (Server → Client):**

- Server က Client ဆီက FIN ကို ရရှိကြောင်း **ACK** ပြန်ပို့ပါတယ်။ (ဒီအချိန်မှာ Server က သူ့ဘက်က ပို့စရာရှိတာတွေကို အပြီးသတ် ပို့ဆောင်နေဆဲ ဖြစ်နိုင်ပါတယ်)။

**FIN, ACK (Server → Client):**

- Server ဘက်ကလည်း ပို့စရာရှိတာ အားလုံးပြီးသွားပြီဆိုရင် ချိတ်ဆက်မှုကို ပိတ်ဖို့အတွက် သူ့ရဲ့ ကိုယ်ပိုင် **FIN** flag ကို Client ဆီ ပြန်ပို့ပါတယ်။

**ACK (Client → Server):**

- နောက်ဆုံးမှာတော့ Client က Server ရဲ့ FIN ကို လက်ခံရရှိကြောင်း **ACK** ပြန်ပို့ပြီး ချိတ်ဆက်မှုကို လုံးဝ အဆုံးသတ်လိုက်ပါတယ်။

### FTP

**FTP** ဆိုတာ ကွန်ပျူတာတွေကြားမှာ ဖိုင်တွေကို မြန်မြန်ဆန်ဆန် ပေးပို့ဖို့အတွက် အသုံးပြုတဲ့ **Application Layer** protocol တစ်ခုဖြစ်ပါတယ်။ 

ပုံမှန် protocol အများစုက Socket တစ်ခုတည်းမှာ ဒေတာနဲ့ Command တွေကို ပို့ဆောင်ပေးပို့ကြပေမယ့် FTP ကတော့ Port (၂) ခုကို တစ်ပြိုင်နက်အသုံးပြုတာကြောင့် ထူးခြားပါတယ်။

- **Port 21 (Command Channel):** FTP session တစ်ခုလုံးကို ထိန်းချုပ်ဖို့အတွက် Command တွေထုတ်ပေးရာမှာ သုံးပါတယ်။
- **Port 20 (Data Channel):** ဖိုင်ဒေတာတွေကို အမှန်တကယ် ပေးပို့တဲ့နေရာမှာ သုံးပါတယ်။

FTP ဟာ ကွန်ရက်အခြေအနေပေါ်မူတည်ပြီး အောက်ပါ Mode နှစ်မျိုးနဲ့ အလုပ်လုပ်နိုင်ပါတယ်-

1. **Active Mode (Default):**
   - Server က Client ဆီကနေ **PORT** command ကို စောင့်ကြည့်ပါတယ်။
   - Client က ဒေတာပို့ဖို့ ဘယ် Port ကို သုံးမယ်ဆိုတာကို Server ဆီ အကြောင်းကြားတဲ့ နည်းလမ်းဖြစ်ပါတယ်။
2. **Passive Mode:**
   - **Firewall** သို့မဟုတ် **NAT** တွေရှိနေလို့ Server ကနေ Client ဆီကို တိုက်ရိုက် TCP connection ဆောက်ဖို့ ခက်ခဲတဲ့အခါ သုံးပါတယ်။
   - Client က **PASV** command ကို ပို့ပြီး Server ဆီကနေ ဘယ် IP နဲ့ Port ကို သုံးရမလဲဆိုတဲ့ အဖြေကို စောင့်ဆိုင်းရတဲ့ နည်းလမ်းဖြစ်ပါတယ်။

| **PWD**     | လက်ရှိ မိမိအလုပ်လုပ်နေသည့် Directory လိပ်စာကို ကြည့်ရှုရန်။             |
| **SIZE**    | သတ်မှတ်ထားသော ဖိုင်တစ်ခု၏ ပမာဏ (Size) ကို ကြည့်ရန်။                |
| **RETR**    | FTP Server ထံမှ ဖိုင်ကို ရယူ (Download) ရန်။                       |
| **QUIT**    | FTP Session ကို အဆုံးသတ်ရန်။                                     |

### SMB

**SMB** ဆိုတာ Windows အသုံးပြုတဲ့ လုပ်ငန်းခွင် (Enterprise Environment) တွေမှာ အတွေ့ရဆုံး Protocol တစ်ခုဖြစ်ပြီး၊ ကွန်ရက်အတွင်းမှာရှိတဲ့ စက်အချင်းချင်း ရင်းမြစ်များ (Resources) မျှဝေသုံးစွဲနိုင်အောင် လုပ်ဆောင်ပေးပါတယ်။ ၎င်းဟာ **Connection-oriented** protocol တစ်ခုဖြစ်တဲ့အတွက် အသုံးမပြုခင်မှာ User ဟာ သက်ဆိုင်ရာ Resource ကို သုံးစွဲခွင့် (Permissions) ရှိမရှိ အရင်ဆုံး **Authentication** (စစ်ဆေးအတည်ပြုခြင်း) လုပ်ဆောင်ဖို့ လိုအပ်ပါတယ်။

**အသုံးပြုပုံ (Usage):** အသုံးပြုသူတွေအတွက် Printer များ၊ Shared Drive များနဲ့ Authentication Server များကို လွယ်ကူလျှင်မြန်စွာ အသုံးပြုနိုင်အောင် ကူညီပေးပါတယ်။

**ပို့ဆောင်ရေးစနစ် (Transport Mechanism):**

- **ယခင်က:** UDP port 137 နှင့် 138 တို့ကို သုံးပြီး NetBIOS မှတစ်ဆင့် အလုပ်လုပ်ခဲ့ပါတယ်။
- **လက်ရှိ:** **TCP port 445** ကို တိုက်ရိုက်အသုံးပြုသလို၊ NetBIOS over TCP (Port 139) နှင့် QUIC protocol တို့ကိုပါ အသုံးပြုနိုင်လာပါပြီ။

**လုံခြုံရေးဆိုင်ရာ သတိပြုရန်:** SMB ဟာ အလွန်အသုံးဝင်သလိုပဲ တိုက်ခိုက်သူ (Attackers) တွေအတွက်လည်း အလွန်ဆွဲဆောင်မှုရှိတဲ့ ပစ်မှတ်တစ်ခု ဖြစ်ပါတယ်။

**အလုပ်လုပ်ပုံ:** TCP ကို အခြေခံထားတဲ့အတွက်ကြောင့် ပုံမှန်အတိုင်းပဲ **Three-way handshake** လုပ်ဆောင်ခြင်းနှင့် Packet ရောက်ရှိကြောင်း အတည်ပြုခြင်း (Acknowledging) တို့ကို လုပ်ဆောင်ပါတယ်။

### Network Traffic Analysis 

**၁။ Passive Traffic Capture (အာရုံခံရုံသက်သက် ဖမ်းယူခြင်း)**

Passive နည်းလမ်းသည် Packet များအပေါ် တိုက်ရိုက်သက်ရောက်မှုမရှိစေဘဲ မိတ္တူကူးယူကြည့်ရှုခြင်း ဖြစ်သည်။

- **လုပ်ဆောင်ပုံ:** Packet များကို တိုက်ရိုက်နှောင့်ယှက်ခြင်း သို့မဟုတ် ပြုပြင်ခြင်း မရှိဘဲ မိမိမြင်ရသမျှ ဒေတာများကို မိတ္တူကူးယူခြင်း ဖြစ်သည်။
- **အားသာချက်:** Network ၏ စွမ်းဆောင်ရည်ကို ထိခိုက်မှုမရှိစေဘဲ စောင့်ကြည့်နိုင်သည်။

**၂။ Active Traffic Capture (တိုက်ရိုက်ပါဝင်ပတ်သက်၍ ဖမ်းယူခြင်း)**

Active နည်းလမ်းသည် ပိုမိုနက်ရှိုင်းသော ချဉ်းကပ်မှုဖြစ်ပြီး ၎င်းကို **In-line traffic captures** ဟုလည်း ခေါ်ဝေါ်သည်။

- **လုပ်ဆောင်ပုံ:** Network လမ်းကြောင်းပေါ်တွင် တိုက်ရိုက်ကြားဖြတ်၍ (In-line) ဖမ်းယူခြင်းဖြစ်သောကြောင့် ပိုမိုထိရောက်သော်လည်း လက်တွေ့ကျကျ ကိုင်တွယ်ဖြေရှင်းရန် လိုအပ်သည်။
- **အားသာချက်:** Traffic များကို အချိန်နှင့်တစ်ပြေးညီ ထိန်းချုပ်နိုင်ခြင်း သို့မဟုတ် ပြောင်းလဲနိုင်ခြင်းတို့ ပြုလုပ်နိုင်သည်။

### Descriptive Analysis

**၁။ ပြဿနာကို ဖော်ထုတ်ခြင်း (Identify the Issue)**

ပထမဆုံးအနေနဲ့ လက်ရှိဖြစ်နေတဲ့ ပြဿနာက ဘာလဲဆိုတာကို သိအောင်လုပ်ရပါမယ်။

- ကျူးကျော်ဝင်ရောက်ခံရတာလား (Suspected breach)?
- ကွန်ရက်ချိတ်ဆက်မှု ပြဿနာလား (Networking issue)?

**၂။ နယ်ပယ်နှင့် ရည်မှန်းချက်ကို သတ်မှတ်ခြင်း (Define Scope and Goal)**

မိမိ ဘာကိုရှာဖွေမှာလဲနဲ့ ဘယ်အချိန်ကာလကို စစ်ဆေးမှာလဲဆိုတာကို တိတိကျကျ သတ်မှတ်ရပါမယ်။

- **ရည်မှန်းချက် (Goal):** ဥပမာ - `bad.example.com` ဆီကနေ အန္တရာယ်ရှိတဲ့ ဖိုင်တွေကို စက်အမြောက်အမြားက ဒေါင်းလုဒ်ဆွဲနေတာကို ရှာဖွေခြင်း။
- **အချိန်ကာလ (Time period):** ဥပမာ - လွန်ခဲ့တဲ့ ၄၈ နာရီကနေ အခုကစပြီး နောက်ထပ် ၂ နာရီအထိ။
- **အထောက်အထားများ (Supporting info):** `superbad.exe` သို့မဟုတ် `new-crypto-miner.exe` ကဲ့သို့သော ဖိုင်အမည်များ သို့မဟုတ် အမျိုးအစားများ။

**၃။ ပစ်မှတ်ကို သတ်မှတ်ခြင်း (Define the Target)**

ဘယ်နေရာမှာ ရှာရမလဲဆိုတာကို သတ်မှတ်ခြင်း ဖြစ်ပါတယ်။

- **ကွန်ရက် (Network):** `192.168.100.0/24` network segment အတွင်းရှိ စက်များ။
- **အသုံးပြုသော Protocol များ:** HTTP နှင့် FTP။

### **Diagnostic Analysis**

Descriptive Analysis လိုပဲ အတိတ်မှာ ဖြစ်ပျက်ခဲ့တာတွေကို ပြန်ကြည့်တာမှန်ပေမယ့်၊ အဓိက ကွာခြားချက်ကတော့ "ဘာကြောင့် ဖြစ်ရတာလဲ" ဆိုတဲ့ အဖြေကို ရှာဖွေဖော်ထုတ်ဖို့ ကြိုးစားတာ ဖြစ်ပါတယ်။

**၁။ Network Traffic ကို ဖမ်းယူခြင်း (Capture)**

ပြဿနာရဲ့ အရင်းအမြစ်ကို ရှာဖို့အတွက် လိုအပ်တဲ့ ဒေတာတွေကို အရင်စုဆောင်းရပါမယ်။

- `192.168.100.0/24` network segment မှာရှိတဲ့ Link တစ်ခုမှာ ချိတ်ဆက်ပြီး Live traffic ကို ဖမ်းယူပါ။ ရည်ရွယ်ချက်ကတော့ ကူးပြောင်းနေတဲ့ အန္တရာယ်ရှိ executable ဖိုင်တွေကို လက်ရဖမ်းမိဖို့ဖြစ်ပါတယ်။
- အတိတ်က ဖြစ်ရပ်တွေကို စစ်ဆေးဖို့အတွက် SIEM (ဥပမာ- Splunk, ELK) ထဲကနေ PCAP ဒါမှမဟုတ် Netflow ဒေတာတွေကို ထုတ်ယူဖို့ Admin ကို အကူအညီတောင်းပါ။

**၂။ လိုအပ်သော Traffic အစိတ်အပိုင်းများကို ခွဲထုတ်ခြင်း (Filtering)**

ဒေတာတွေ အများကြီးထဲကနေ ကိုယ်နဲ့မဆိုင်တာတွေကို ဖယ်ထုတ်ပစ်ရပါမယ်။

- မိမိတို့ရဲ့ **Baseline** (ပုံမှန်အခြေအနေ) နဲ့ ကိုက်ညီနေတဲ့ ပုံမှန် Traffic တွေကို Filter သုံးပြီး ဖယ်ထုတ်ပါ။
- စုံစမ်းစစ်ဆေးမှုနဲ့ သက်ဆိုင်တဲ့ HTTP, FTP traffic တွေနဲ့ အန္တရာယ်ရှိဖိုင်တွေကို တောင်းဆိုထားတဲ့ **GET request** တွေကိုပဲ ချန်လှပ်ထားပါ။

**၃။ ဖမ်းယူရရှိထားသော Traffic ကို နားလည်အောင် ခွဲခြမ်းစိတ်ဖြာခြင်း (Understanding)**

ဆူညံသံ (Noise) တွေကို ဖယ်ထုတ်ပြီးပြီဆိုရင် ပစ်မှတ်ကို စတင်ရှာဖွေပါမယ်။

- **FTP အတွက်:** `ftp-data` ကို filter လုပ်ပြီး ဘယ်ဖိုင်တွေ ပေးပို့ခဲ့သလဲဆိုတာ ကြည့်ပါ။ လိုအပ်ရင် အဲဒီဖိုင်တွေကို ပြန်လည်ပေါင်းစည်း (Reconstruct) ကြည့်ပါ။
- **HTTP အတွက်:** `http.request.method == "GET"` ကို သုံးပြီး မိမိတို့ ရှာဖွေနေတဲ့ ဖိုင်အမည်တွေနဲ့ ကိုက်ညီတဲ့ တောင်းဆိုမှုတွေကို စစ်ဆေးပါ။
- ဒီနည်းလမ်းကနေတစ်ဆင့် ဘယ်သူတွေက အဲဒီဖိုင်ကို ရယူသွားသလဲနဲ့ ကွန်ရက်အတွင်းမှာ တခြားဘယ်စက်တွေကို ပြန့်နှံ့သွားသလဲဆိုတာကို သိရှိနိုင်ပါတယ်။

### Predictive Analysis

**Predictive Analysis** ဆိုတာ အတိတ်ကဖြစ်ရပ်တွေ (Historical data) နဲ့ လက်ရှိအခြေအနေတွေကို သုံးသပ်ပြီး နောင်တစ်ချိန်မှာ ဘာတွေဖြစ်လာနိုင်မလဲဆိုတာကို ကြိုတင်ခန့်မှန်းတဲ့ လုပ်ငန်းစဉ်ဖြစ်ပါတယ်။ အဓိက ရည်ရွယ်ချက်ကတော့ ပုံမှန်အခြေအနေကနေ သွေဖည်နေတဲ့ အချက်တွေကို စောစီးစွာ သိရှိနိုင်ဖို့နဲ့ အနာဂတ်မှာ ဖြစ်လာနိုင်တဲ့ ဘေးအန္တရာယ်တွေကို တိကျစွာ ခန့်မှန်းနိုင်ဖို့ ဖြစ်ပါတယ်။

**၁။ တွေ့ရှိချက်များကို မှတ်တမ်းတင်ခြင်း (Note-taking and Mind mapping)**

စုံစမ်းစစ်ဆေးမှုတစ်လျှောက် တွေ့ရှိသမျှအားလုံးကို သေသေချာချာ မှတ်တမ်းတင်ထားဖို့ လိုအပ်ပါတယ်။ ဒါမှသာ အနာဂတ်အတွက် ပုံစံတူ (Pattern) တွေကို ရှာဖွေနိုင်မှာပါ။

- **Timeframes:** Traffic ဖမ်းယူခဲ့တဲ့ အချိန်ကာလများ။
- **Suspicious Hosts:** ကွန်ရက်အတွင်းရှိ သံသယဖြစ်ဖွယ် စက်များ။
- **Conversations:** အန္တရာယ်ရှိဖိုင်များ ပါဝင်နေတဲ့ ဆက်သွယ်မှုများ (အချိန်နှင့် Packet နံပါတ်များ အပါအဝင်)။

**၂။ ခွဲခြမ်းစိတ်ဖြာမှု အနှစ်ချုပ် (Summary of the Analysis)**

မိမိတွေ့ရှိခဲ့တဲ့ အချက်အလက်တွေကို အထက်လူကြီးတွေ ဆုံးဖြတ်ချက်ချနိုင်အောင် ရှင်းလင်းပြတ်သားစွာ အစီရင်ခံစာ တင်ပြရပါမယ်။

- **ရည်ရွယ်ချက်:** သံသယဖြစ်ဖွယ် စက်တွေကို ကွန်ရက်ကနေ ခေတ္တဖြတ်တောက် (Quarantine) မလား ဒါမှမဟုတ် ပိုမိုကြီးမားတဲ့ Incident Response တွေ လုပ်မလားဆိုတာကို ဆုံးဖြတ်ဖို့ ဖြစ်ပါတယ်။

### Prescriptive Analysis

Analysis လုပ်ငန်းစဉ်အားလုံးရဲ့ နောက်ဆုံးပန်းတိုင်ဖြစ်ပြီး "ပြဿနာကို ဖြေရှင်းဖို့နဲ့ နောက်တစ်ကြိမ် မဖြစ်အောင် ဘယ်လိုတားဆီးမလဲ" ဆိုတဲ့ လက်တွေ့လုပ်ဆောင်ရမယ့် အဆင့်တွေကို ဖော်ထုတ်တာ ဖြစ်ပါတယ်။အရှေ့မှာ လုပ်ခဲ့တဲ့ Descriptive, Diagnostic နဲ့ Predictive analysis ရလဒ်တွေကို အခြေခံပြီး အကောင်းဆုံး ဆုံးဖြတ်ချက်တွေ ချမှတ်တဲ့ အဆင့်လို့ ဆိုနိုင်ပါတယ်။

**ပြဿနာကို သတ်မှတ်ခြင်း:** (ဥပမာ- Breach လား? Network error လား?)

**နယ်ပယ်သတ်မှတ်ခြင်း:** (ဘယ်အချိန်? ဘယ်စက်တွေ? ဘယ်ဖိုင်တွေ?)

**ပစ်မှတ်သတ်မှတ်ခြင်း:** (Net, Host, Protocol - ဥပမာ HTTP/FTP)

**Traffic ဖမ်းယူခြင်း:** (Live capture သို့မဟုတ် PCAP/Netflow ထုတ်ယူခြင်း)

**Filtering လုပ်ခြင်း:** (Baseline နှင့် မဆိုင်သည်များကို ဖယ်ထုတ်ခြင်း)

**နားလည်အောင် ခွဲခြမ်းစိတ်ဖြာခြင်း:** (GET request များ၊ `ftp-data` များကို စစ်ဆေးခြင်း)

**မှတ်တမ်းတင်ခြင်း:** (Timestamp နှင့် Packet နံပါတ်များ မှတ်သားခြင်း)

**အနှစ်ချုပ်တင်ပြခြင်း:** (ဆုံးဖြတ်ချက်ချနိုင်ရန် ရှင်းလင်းစွာ အစီရင်ခံခြင်း)

---

### Key Components of an Effective Analysis 

**၁။ ကိုယ့်ရဲ့ Environment ကို ကောင်းကောင်းသိပါ (Know your environment)**

ဒါက အခြေခံအကျဆုံးအချက်ပါ။ ကိုယ့် Network ထဲမှာ ဘယ်လို Device တွေရှိလဲ၊ IP Address တွေက ဘာတွေလဲဆိုတာ မသိရင် ပုံမှန်မဟုတ်တဲ့ (Rogue) စက်တွေကို ခွဲခြားနိုင်မှာ မဟုတ်ပါဘူး။

- **Asset Inventories:** စက်ပစ္စည်းစာရင်းများ ထားရှိပါ။
- **Network Maps:** ကွန်ရက်ချိတ်ဆက်ပုံ မြေပုံများ ရှိပါစေ။
- ဒါမှသာ ခွဲခြမ်းစိတ်ဖြာတဲ့အခါမှာ ဘယ်စက်က ကိုယ့်လူ၊ ဘယ်စက်က သူစိမ်းဆိုတာကို ချက်ချင်း သိနိုင်မှာ ဖြစ်ပါတယ်။

------

**၂။ နေရာချထားမှုသည် အဓိကဖြစ်သည် (Placement is Key)**

Traffic ဖမ်းယူမယ့်နေရာ (Capture Tool ထားမယ့်နေရာ) က အလွန်အရေးကြီးပါတယ်။ ပြဿနာဖြစ်နေတဲ့နေရာနဲ့ အနီးဆုံးမှာ ထားရှိခြင်းက အကောင်းဆုံးဖြစ်ပါတယ်။

- **Internet မှလာသော Traffic:** အဝင် Link (Inbound links) တွေကို စောင့်ကြည့်ခြင်းက အကောင်းဆုံးပါ။ ဒါမှသာ အပြင်က ဘာတွေဝင်လာလဲဆိုတာ ပုံရိပ်အပြည့်အစုံကို မြင်ရမှာဖြစ်ပါတယ်။
- **Internal Network ပြဿနာ:** စက်တစ်လုံးတည်းမှာ ပြဿနာဖြစ်နေတယ်ဆိုရင် အဲဒီစက်ရှိတဲ့ Network Segment ထဲမှာပဲ Capture Tool ကို ထားပြီး စစ်ဆေးပါ။

------

**၃။ ဇွဲရှိမှု (Persistence)**

Traffic Analysis လုပ်တဲ့အခါ ပြဿနာကို ချက်ချင်း ရှာမတွေ့နိုင်ပါဘူး။ တိုက်ခိုက်သူတွေက သူတို့ရဲ့ **Command and Control (C2)** ဆာဗာတွေဆီကို အချိန်အကြာကြီး (ဥပမာ- နာရီပေါင်းများစွာ သို့မဟုတ် တစ်ရက်မှတစ်ကြိမ်) မှတစ်ဆင့် လှမ်းချိတ်တာမျိုး လုပ်တတ်ကြပါတယ်။

- **စိတ်ရှည်ပါ:** ပထမတစ်ခေါက်မှာ ဖမ်းမမိခဲ့ရင်တောင် နောက်တစ်ကြိမ် ထပ်ပေါ်လာမယ့်အချိန်ကို စောင့်ကြည့်ရပါမယ်။
- **အရေးကြီးပုံ:** ဒီလို စိတ်ရှည်စောင့်ကြည့်ခြင်းက တိုက်ခိုက်သူကို အချိန်မီတားဆီးနိုင်ဖို့နဲ့ Ransomware ကဲ့သို့ ကြီးမားတဲ့ တိုက်ခိုက်မှုတွေ မဖြစ်အောင် ကာကွယ်ပေးနိုင်ဖို့ အဆုံးအဖြတ်ပေးနိုင်ပါတယ်။

---

### Analysis Approach

**၁။ အသုံးများသော Protocol များမှ စတင်ပါ (Start with Standard Protocols)**

တိုက်ခိုက်မှုအများစုဟာ အင်တာနက်ကနေလာတာဖြစ်လို့ **HTTP/HTTPS, FTP, E-mail (SMTP/IMAP)** စတဲ့ Protocol တွေကို အရင်စစ်ဆေးပါ။

- ပုံမှန်မဟုတ်တဲ့ Traffic တွေကို ဖယ်ထုတ်ပြီးရင် **SSH, RDP, Telnet** စတဲ့ ချိတ်ဆက်မှုတွေကို ဆက်ကြည့်ပါ။
- **သတိပြုရန်:** မိမိအဖွဲ့အစည်းရဲ့ Security Policy က အပြင်ကနေ RDP လှမ်းဝင်တာကို ခွင့်ပြုသလား? Telnet သုံးခွင့်ပေးသလား? ဆိုတာကို သိထားဖို့ လိုပါတယ်။

------

**၂။ ပုံစံတူများကို ရှာဖွေပါ (Look for Patterns)**

စက်တစ်လုံးဟာ နေ့စဉ် တစ်ချိန်တည်းမှာပဲ အင်တာနက်က နေရာတစ်ခုခုကို လှမ်းချိတ်နေသလား?

- ဒါဟာ **Command and Control (C2)** ဆာဗာနဲ့ ချိတ်ဆက်နေတဲ့ ပုံစံမျိုး ဖြစ်နိုင်ပါတယ်။
- ဒီလို ပုံသေကားကျ ဖြစ်နေတဲ့ (Periodic/Beaconing) Traffic တွေကို ရှာဖွေခြင်းဖြင့် တိုက်ခိုက်သူကို ဖော်ထုတ်နိုင်ပါတယ်။

------

**၃။ စက်အချင်းချင်း ဆက်သွယ်မှုကို စစ်ဆေးပါ (Host-to-Host Traffic)**

ပုံမှန်အားဖြင့် ရုံးခန်းတစ်ခုထဲက User PC ချင်း တိုက်ရိုက်စကားပြောလေ့ မရှိပါဘူး။ စက်တွေဟာ IP ယူဖို့၊ DNS မေးဖို့ ဒါမှမဟုတ် Server တွေနဲ့ ချိတ်ဆက်ဖို့အတွက်ပဲ အလုပ်လုပ်ကြတာပါ။

- အကယ်၍ User PC နှစ်လုံး တိုက်ရိုက် ချိတ်ဆက်နေတာ မြင်ရင် သံသယရှိရပါမယ်။ ဒါဟာ **Lateral Movement** (တိုက်ခိုက်သူက ကွန်ရက်ထဲမှာ တစ်စက်ကနေ တစ်စက် ကူးစက်နေခြင်း) ဖြစ်နိုင်ပါတယ်။

------

**၄။ ထူးခြားသော ဖြစ်ရပ်များကို အာရုံစိုက်ပါ (Unique Events)**

ပုံမှန်မဟုတ်တဲ့ အပြောင်းအလဲတွေကို သတိထားပါ။

- **User-Agent:** မရင်းနှီးတဲ့ Browser သို့မဟုတ် Application နာမည်တွေ ပါနေသလား?
- **Random Ports:** စက်တစ်လုံးမှာ ပုံမှန်မဟုတ်တဲ့ Port တစ်ခု ပွင့်နေတာ ဒါမှမဟုတ် ခဏလေး ချိတ်ဆက်သွားတာမျိုး ရှိသလား?
- **Behavior Change:** တစ်နေ့ကို ဝဘ်ဆိုဒ်တစ်ခုဆီ ၁၀ ကြိမ်သွားနေကျ စက်ကနေ ဒီနေ့ ၁ ကြိမ်ပဲ သွားတော့တာမျိုးဟာလည်း စုံစမ်းဖို့ လိုအပ်တဲ့ အချက်ဖြစ်ပါတယ်။

------

**၅။ အကူအညီတောင်းရန် ဝန်မလေးပါနှင့် (Ask for Help)**

Packet တွေကို အကြာကြီးကြည့်နေတဲ့အခါ မျက်စိလျှံပြီး အရေးကြီးတာတွေကို လွတ်သွားနိုင်ပါတယ်။ တခြားလုပ်ဖော်ကိုင်ဖက်တစ်ဦးကို ဝိုင်းကြည့်ခိုင်းခြင်းက မမြင်ရတဲ့ အမှားတွေကို ရှာတွေ့ဖို့ အများကြီး အထောက်အကူပြုပါတယ်။
